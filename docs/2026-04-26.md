## 2. 新增的关键零件

### 2.1 `benchmark/hermes_direct_runner.py` (457 行)

这是上次报告点名的"最大瓶颈"，现已实现。要点：

- **直接 in-process 调用** `hermes_v-0-10-0/run_agent.py:AIAgent`，把 `harness_config` 翻译成 `max_iterations`、`enabled_toolsets`、`disabled_toolsets`、`provider`、`base_url` 等构造参数（`hermes_direct_runner.py:369`）
- **三层 contextmanager**：`_temporary_env`、`_temporary_cwd`、`_temporary_tool_schema_gate`，确保 benchmark 跑完后 hermes 全局状态干净
- **Per-tool allow/deny 门控**（`hermes_direct_runner.py:171`）：通过 monkey-patch `model_tools.get_tool_definitions`，在不改 hermes 源码的前提下做比 toolset 更细的工具裁剪——这正是文档 §3.2 说的 "Action Gating" 元素
- **strict_runtime 守卫**（`hermes_direct_runner.py:316`）：显式 `base_url` 必须配对显式 api key，否则 runner 主动报错而不是悄悄 fallback——已经在防御一类真实的配置误用
- **Stdout/stderr 阶段化捕获**：`agent_init` / `run_conversation` / `agent_close` 三段分别录制，避免 hermes 内部日志污染 trajectory
- **Trajectory 中间格式正确填齐**：`tool_calls`、`final_message`、`usage`、`stop_reason`、`runtime`，与 `metrics.py` 的契约一致

### 2.2 第一组 Action Gating ablation 配置

`configs/` 现有 3 个 yaml：

| 配置 | 工具暴露 | 对照含义 |
|---|---|---|
| `baseline.yaml` | `[terminal, file]`（toolset 级） | 双工具集，无 per-tool 限制 |
| `action_file_only.yaml` | `[file]` + allow `[read_file, search_files]` | 只读文件操作，禁 shell |
| `action_terminal_only.yaml` | `[terminal]` + allow `[terminal, process]` | 只用 shell |

这是**文档 §10 第三阶段"单因素 ablation"的第一次落地**，且只动 Action Gating 这一个轴（Observation/Transition 两段保持一致），对照学严谨。

## 5. 推荐的下一步（按 ROI 重排）

1. **跑完 terminal_only**——一条命令就能把 Action Gating 三元组补齐；预计能直接证伪/证实"terminal 比 file 工具更快"或"terminal 更易工作目录错乱"
2. **写 `scripts/aggregate_results.py`**——读 `results/*.jsonl`，按 (family, config) 输出 pass rate / latency p50p95 / token mean。这是后续所有 ablation 的基础设施
3. **加 seed 数到 5–7**——目前 30 records 太薄，Q=100% 的差别可能被 seed 噪声掩盖
4. **填 F2 最小子集**——5 个 SWE-bench Lite 实例足够开始测"代码修复型 vs 代码理解型"假设
5. **再开一个 Transition Control 轴的 ablation**——比如 `transition_max_iter_10.yaml` vs `transition_max_iter_60.yaml`，复用现有 baseline 框架就能跑
6. **最后再写 sandbox**——只有要跑 F4 时才阻塞
