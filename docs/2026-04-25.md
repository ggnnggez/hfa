## 3. 关键缺口（按阻塞优先级）

1. **没有真 runner**（最大瓶颈）。`stub_runner.py:11` 返回空 trajectory，所以全栈即使能跑也产不出有意义的 Q/L/C/R。`HermesDirectRunner` 需要把 `baseline.yaml` 里的 22 个 knob 翻译成 `hermes_v-0-10-0/run_agent.py` 的 `IterationBudget`、toolset 选择、`pre_tool_call` hooks 等实际入口——这是文档第二阶段往第三阶段过渡的硬门槛。
2. **F2/F3 完全空缺**。任务文件 0 行，oracle 是占位符。这意味着目前的实验只能覆盖文档分类里的两族（检索型 + 高风险执行型），完全测不到"长链路任务依赖 Transition Control"这条假设——而那是文档 §6 给出的核心 ablation 论点之一。
3. **只有 baseline，没有 ablation 配置**。文档 §10 第三阶段要求"单因素 ablation → 双因素 → factorial"，但 `configs/` 目录下只有 1 个 yaml。需要至少一组 `obs_*.yaml`、`act_*.yaml`、`trans_*.yaml` 的变体生成器（推荐脚本化生成而不是手写每个文件）。
4. **没有结果聚合 / Pareto 分析层**。文档 §5.2 推荐方式 A（Pareto）+ 方式 C（约束优化）作为初期主路线，但仓库里没有从 JSONL 算 frontier、按 family 汇总均值/方差、做 ablation 对比的代码。
5. **execution 段的 `env: docker` 只是字符串**。没有 sandbox 实现；F4 的 setup 命令（如 `printf > data.txt`、`mkdir`）需要在隔离 workspace 里运行，否则会污染 `/home/nan/c_project/hfa`。

## 4. 与文档对齐度的细评

**对齐得很好**：
- 三类控制（O/A/T）从 taxonomy 到 baseline.yaml 名字一一对应，便于按类别打 ablation 标签。
- Q/L/C/R 拆成 oracle-pass / detail / wall_clock / 三种 token / 5 种 risk 信号，与 §5.1 的"先保留多目标结构，不急着压成单标量"一致。
- Family 选择覆盖了 §6 的"不同 family 对三类控制依赖结构不同"的核心假设——只是 F2/F3 还没填。

**还没对齐**：
- 文档 §7 把 knob 进一步参数化（如 prefetch top-k、compression target ratio），但 baseline.yaml 里这些参数全是 `null` 或 `false`。要做有意义的 ablation，得为每个 knob 给出 2–3 个候选值。
- 文档 §8 的"互补 / 替代 / 冲突"关系完全没在配置层面表达。当前没机制能阻止生成"过宽工具 + 严 approval"这种内部冲突的组合，未来 factorial search 时会浪费算力。
- 没有 task feature 提取（文档 §9 / §11 的 `φ(x)`），所以离 routing policy 还有两步距离。
